# 竞品对标UI走查用例生成规范

## 目标要求
当用户请求生成竞品对标UI走查用例时，直接创建CSV格式的用例文档，并将CSV文件存放在项目的`output`目录下。本规范专注于竞品对标场景，帮助产品在对标前进行全面自查，避免常见问题。

## CSV格式规范

### 文件命名规范
- 文件名格式：`{产品名称}-竞品对标UI走查用例-{序号}.csv`
- 存放目录：`output`（项目根目录下的output文件夹）
- 示例：`output/智算平台-竞品对标UI走查用例-1.csv`

### CSV文件结构
CSV文件包含一个UI走查用例清单，使用UTF-8编码，以逗号作为分隔符。

### UI走查用例清单列结构
| 列名 | 说明 | 必填 |
|------|------|------|
| 用例编号 | CP-TC001、CP-TC002格式（CP=Competitive Product） | 是 |
| 页面/模块 | 所属页面或功能模块名称 | 是 |
| 检查点 | 具体的功能或UI元素 | 是 |
| 设计原则 | 对应的竞品对标设计原则 | 是 |
| 检查项 | 描述具体的检查内容 | 是 |
| 优先级 | 高/中/低 | 是 |
| 预期结果/标准 | 符合竞品对标要求的具体标准 | 是 |
| 实际结果 | 实际执行后的结果 | 否 |
| 截图/备注 | 用于辅助说明的截图或备注信息 | 否 |

### 优先级划分规范

**重要说明**：优先级基于**检查的重要性**，而非问题的严重性。即：如果不检查这一项，在竞品对标中可能导致什么后果。

#### 高优先级（70-80%）- 必须检查
**判断标准**：如果不检查，可能在竞品对标中被扣分

**检查内容**：
- **系统可靠性（最高优先级）**
  - 所有异常场景的处理（报错、失败、超时）
  - 核心功能的可用性
  - 页面响应速度（<3秒）
  - 输入校验的完整性（特殊字符、边界值）

- **信息清晰性（高优先级）**
  - 关键信息的完整性（费用、到期、限制说明）
  - 操作提示的清晰度

- **跳转准确性（高优先级）**
  - 所有跳转是否准确到达目标页面
  - 链接是否有效

#### 中优先级（15-25%）- 应该检查
**判断标准**：影响用户体验，可能在竞品对比时被发现

**检查内容**：
- 帮助文档与产品的一致性
- 同类信息的展示统一性
- 语言的统一性（避免中英文混合）
- 批量操作的支持

#### 低优先级（5-10%）- 可选检查
**判断标准**：优化建议，不太可能影响对标得分

**检查内容**：
- 视觉细节优化
- 极端边缘场景
- 锦上添花的功能
## 竞品对标十大设计原则
基于123个竞品对标问题分析，提炼出以下十大设计原则，用于指导UI走查用例生成：

### 原则1: 异常处理完备性
**核心要求**：系统必须具备完善的异常处理机制，确保所有异常情况都能被捕获、处理并友好提示。

**检查要点**：
- 所有接口调用都有异常捕获
- 报错信息清晰说明原因和解决方法
- 异常情况提前判断和提示
- 避免出现500、400等技术性错误
- 长时间操作有超时处理
- 失败操作可重试或回滚

**反面案例**：
```
❌ 创建失败，无明确原因
❌ 接口报500，服务器内部错误
❌ 偶现报错"通讯异常"
```

**正面标准**：
```
✅ 创建失败：资源不足，请稍后重试或联系管理员
✅ 操作超时：网络连接不稳定，已自动保存，可继续操作
✅ 参数错误：请输入1-100之间的整数
```

### 原则2: 信息提示完整性
**核心要求**：所有关键信息、操作提示、费用说明必须完整、清晰、友好地展示给用户。

**检查要点**：
- 费用、价格信息明确说明
- 到期、续订等关键时间节点有提示
- 操作前提和限制条件清晰说明
- 提示语避免技术术语和代码语言（如null）
- 必填项、可选项明确标识
- 操作结果和影响明确告知

**反面案例**：
```
❌ 提示语中有null代码语言
❌ 续订时未说明续订费用
❌ 到期前没有提示
❌ 子网提示信息不清晰
```

**正面标准**：
```
✅ 续订费用：￥XXX元/月，续订后将于XXXX年XX月XX日到期
✅ 您的资源将于3天后到期，请及时续订以免影响使用
✅ 裸金属子网：用于裸金属节点的网络配置，请选择已创建的子网
```

### 原则3: 功能可用性保障
**核心要求**：所有功能必须稳定可用，失效功能应及时修复或下架，不可用功能应禁用并说明原因。

**检查要点**：
- 核心功能稳定可用
- 失效功能及时修复或下架
- 不可用功能置灰并说明原因
- 功能限制提前告知
- 依赖条件未满足时禁用功能
- 功能状态实时更新

**反面案例**：
```
❌ 代码补全功能失效
❌ 无法进行问答
❌ 刷新功能异常
❌ 存在多款智能体无法使用
```

**正面标准**：
```
✅ 该功能需要先配置XX，点击前往配置
✅ 当前版本暂不支持此功能，敬请期待
✅ 该功能维护中，预计XX时间恢复
```

### 原则4: 文档同步一致性
**核心要求**：帮助文档、产品介绍必须与产品实际功能保持同步，及时更新，链接有效。

**检查要点**：
- 帮助文档与产品功能同步更新
- 文档截图与实际界面一致
- 所有链接有效可访问
- 产品介绍信息准确最新
- 版本更新后文档同步更新
- 操作步骤与实际流程一致

### 原则5: 响应速度优化
**核心要求**：系统响应速度必须快速，页面加载、刷新、操作响应时间符合用户预期。

**检查要点**：
- 页面首次加载时间<3秒
- 刷新操作响应及时
- 长时间操作有进度提示
- 配额、状态更新及时
- 避免一直显示加载中
- 优化大数据量加载

### 原则6: 跳转准确性
**核心要求**：所有跳转必须准确到达目标页面，避免跳转错误、需要二次操作或链接失效。

**检查要点**：
- 跳转目标准确无误
- 跳转后直达目标功能，无需二次操作
- 默认跳转符合用户预期
- 跳转链接有效可访问
- 跨产品跳转携带必要参数
- 返回操作保留用户状态

### 原则7: 信息一致性
**核心要求**：同类信息的展示方式、格式、逻辑必须保持一致，避免用户困惑。

**检查要点**：
- 费用显示逻辑统一
- 价格信息格式一致
- 单词拼写统一
- 单位显示规则统一
- 同类操作入口位置一致
- 视觉样式保持一致

### 原则8: 输入校验完整性
**核心要求**：所有用户输入必须进行完整校验，过滤特殊字符，限制输入范围，提供友好提示。

**检查要点**：
- 特殊字符过滤（%、&、<、>等）
- 输入长度限制
- 输入格式校验
- 必填项校验
- 参数有效性校验
- 校验失败友好提示

### 原则9: 语言统一性
**核心要求**：产品界面、提示信息、文档必须使用统一语言，避免中英文混合。

**检查要点**：
- 界面文案统一使用中文
- 提示信息避免中英文混合
- 按钮文字统一语言
- 专业术语提供中文说明
- 错误信息使用中文
- 帮助文档语言统一

### 原则10: 操作高效性
**核心要求**：支持批量操作，减少重复步骤，提升操作效率，达到或超越竞品水平。

**检查要点**：
- 支持批量上传、删除、修改
- 支持多文件并行处理
- 避免重复操作
- 提供快捷操作方式
- 操作后停留在合理位置
- 自动保存用户状态
## 竞品对标高频问题类型

基于123个竞品对标问题分析，以下是需要重点检查的高频问题类型：

### 1. 系统报错和异常处理（占比24.4%）
- 系统频繁报错、异常、失败
- 报错信息不友好，未说明原因
- 异常情况未提前判断和提示
- 接口返回500、400等错误

### 2. 提示信息不清晰或缺失（占比17.1%）
- 提示语不友好，包含代码语言
- 缺少必要的操作提示和说明
- 费用、价格信息未说明
- 到期、续订等关键信息无提示

### 3. 功能无法正常使用（占比14.6%）
- 功能失效、无法触发
- 操作无法完成
- 页面一直加载、显示异常
- 功能不可用但未禁用

### 4. 帮助文档与产品不一致（占比8.1%）
- 帮助文档未及时更新
- 文档内容与实际功能不符
- 文档图片过时
- 链接失效、页面不存在

### 5. 加载和刷新问题（占比6.5%）
- 页面加载速度慢
- 刷新功能异常
- 一直显示加载中
- 配额更新延迟

### 6. 跳转逻辑问题（占比6.5%）
- 跳转目标不正确
- 跳转后需要二次操作
- 跳转链接失效
- 默认跳转不符合预期

## 生成要求

### 必须要求
1. **直接生成**：直接使用`write`工具创建CSV格式UI走查用例文件
2. **目录位置**：CSV文件必须存放在项目的`output`目录下
3. **数据完整性**：CSV文件必须包含所有必要的列和数据
4. **格式规范**：严格按照CSV格式要求创建文件，UTF-8编码，逗号分隔
5. **原则覆盖**：确保十大设计原则都有对应的检查用例
6. **高频问题覆盖**：重点覆盖六大高频问题类型

### 数据生成规则
- 根据产品功能和模块结构生成CSV UI走查用例数据
- 页面/模块根据产品的功能模块结构确定
- 检查点基于具体的功能或UI元素
- 设计原则从十大竞品对标设计原则中选择最相关的一个
- 用例编号格式：CP-TC001、CP-TC002、CP-TC003按顺序递增
- 检查项描述清晰、具体，便于执行验证
- 预期结果/标准明确且可验证，参考正面标准示例
- CSV文件中包含换行和特殊字符的内容需要用双引号包围并转义处理

### 优先级分配规则
- **高优先级**：涉及系统可靠性（报错、异常、失败）、信息清晰性（费用、到期提示）、功能可用性的检查项
- **中优先级**：涉及响应速度、跳转准确性、信息一致性的检查项
- **低优先级**：涉及语言统一性、操作高效性的优化类检查项

## 竞品对标走查方法

### 方式一：按原则走查
按照十大设计原则，逐条检查每个原则在产品中的应用情况，确保所有原则都被检查到位。

**检查顺序**：
1. 异常处理完备性（最高优先级，占比65.9%）
2. 信息提示完整性（第二优先级，占比19.5%）
3. 功能可用性保障
4. 文档同步一致性
5. 响应速度优化
6. 跳转准确性
7. 信息一致性
8. 输入校验完整性
9. 语言统一性
10. 操作高效性

### 方式二：按问题类型走查
按照高频问题类型，重点检查容易出现问题的场景。

### 方式三：按模块走查
按照产品的功能模块，系统性地检查每个模块的竞品对标问题。

## 工作流程（5步法）
1. **理解产品功能**：熟悉产品的功能模块、业务流程和用户场景
2. **应用设计原则**：结合十大竞品对标设计原则，系统性地设计检查点
3. **覆盖高频问题**：确保六大高频问题类型都有对应的检查用例
4. **准备数据**：将检查项整理为CSV格式数据，确保覆盖所有关键场景
5. **创建CSV**：直接调用`write`工具创建CSV文件，并保存到项目的`output`目录

## 验收标准

### 基本质量要求
1. 文件能正常用Excel/文本编辑器打开，无格式错误
2. 包含完整的UI走查用例信息
3. 格式规范，易于阅读和编辑
4. 支持后续的竞品对标走查执行跟踪

### 竞品对标用例质量标准
- **原则覆盖性**：十大设计原则都有对应的检查用例
- **问题覆盖性**：六大高频问题类型都有对应的检查用例
- **可验证性**：预期结果/标准必须是具体且可验证的
- **清晰性**：检查项和预期结果描述清晰，无歧义
- **完整性**：UI走查用例包含执行所需的所有信息
- **针对性**：用例针对竞品对标场景，避免常见问题

### 覆盖率要求
- **系统可靠性检查**：至少覆盖50%的用例（对应65.9%的问题占比）
- **信息清晰性检查**：至少覆盖20%的用例（对应19.5%的问题占比）
- **其他原则检查**：至少覆盖30%的用例
- **每个功能模块**：至少应用5个以上的设计原则

## 注意事项
1. **CSV格式优先**：严格按照CSV格式输出，不生成Markdown文件
2. **工具依赖**：使用`write`工具进行CSV文件操作
3. **路径规范**：使用正确的绝对路径格式，确保文件正确存放在output目录
4. **数据格式**：检查项描述清晰具体，便于执行验证
5. **编码规范**：使用UTF-8编码，确保中文字符正确显示
6. **字符转义**：CSV文件中包含逗号、双引号、换行符的内容需要用双引号包围并转义
7. **列结构固定**：保持9列结构
8. **覆盖率优先**：优先保证竞品对标问题覆盖率，确保常见问题都被检查
9. **标准引用**：预期结果/标准应参考正面标准示例，而非模糊描述
10. **重点突出**：重点关注系统可靠性（65.9%）和信息清晰性（19.5%）两大问题类型
11. **实用性优先**：用例应该实用、可执行，能够真正帮助发现竞品对标问题
